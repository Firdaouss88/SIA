Architecture fonctionnelle - Configuration des sources de données

ARCHITECTURE FONCTIONNELLE - CONFIGURATION DES SOURCES DE DONNÉES

Les espaces de travail de jeux de données sont spécifiquement créés pour y déposer les jeux de données et uniquement les jeux de données.

Source de données acceptées pour alimenter un jeu de données Power BI​

Pour le développement corporatif​

La seule source de données valides est un comptoir.

Il existe plusieurs type de comptoir de données.

Comptoir de données

  • Produit par EODE, pour les projet affaires.
  • Préfixe de base de données = "DM_"

Comptoir de données projet Fast-Track.

  • Produit par EODE, pour les projets rapide ou temporaire non intégré dans la voute de données.
  • Préfixe de base de données = "DMF_"

Sandbox EODE.

  • Pour les projets d'explorations, de validation et d'évaluation de solution.
  • Préfixe de base de données = "SBX_"

Sandbox d'exploration affaires.

  • Pour les projets de développement en milieu utilisateur.
  • Préfixe de base de données = "SBXU_"

Pour la développement en milieu utilisateur​

En plus des comptoirs, les concepteurs affaires peuvent utiliser Excel et des fichiers plat comme source de données.

Autres source de données, développement​

Dans les cas ou la source de données n'est pas un comptoir, il faut la rendre disponible via un comptoir de données.

Exemple:

Pour explorer le Data Lake, la voute de données brute et la voute affaire, un sandbox sera créer afin de rendre les sources requises disponible aux développeur et/ou concepteurs.

Pour explorer des fichiers plats, il est recommandé de la charger dans un sandbox et d'utiliser ce sandbox comme source pour produire le jeu de données Power BI.

Objectif d'utiliser un comptoir pour consommer les différentes données provenant des couches de l'entrepôts de données.

  • Permet de ne pas données accès au sous-couche de données, autre que la couche des comptoirs.

  • Contrôler les données accessible aux concepteurs via le comptoir.

    • Le concepteur à seulement accès au données présentes dans le comptoir, mais pas à la source originale des données.

  • Permet de limiter et d'encadrer l'accès au données d'une couche de l'entrepôt sans avoir à sécurité cette couche par utilisateur.

    • Le Data Lake peut contenir des centaines de tables provenant de plusieurs sources de données.
    • Un comptoir peut exposer un sous-ensemble du Data Lake pour rendre disponible les tabes d'un sujet données.
    • L'accès sera aux concepteurs sera donné sur le comptoir mais pas sur le Data Lake, limitant automatiquement l'accès aux données à celle exposées dans le comptoir.

  • Personne n'accède directement le Data Lake et la voute (brute et affaire).

    • Le "Cross chaining" des droits dans SQL permet de créer un vue dans un sandbox sur des données du Data Lake et de données accès en lecture à cette vue, dans le sandbox, sans données accès au Data Lake.

  • La gestion des accès utilisateurs se fait sur une seule couche de l'entrepôt.

    • La couche des comptoirs.
    • Aucune autre couche n'est exposer à des utilisateurs autre que les développeurs de l'équipe BI

Voici les différents scénarios possibles:​

La configuration à mettre en place dans le service Power BI dépend de la source de données à gérer.

  • Une source -> un jeu de données (défaut).
    • Une source de donnée pour produire un jeu de données Power BI.
  • Cas Plusieurs sources -> un jeu de données
    • Plusieurs sources de données pour produire un jeu de données Power BI.
  • Une source -> plusieurs jeux de données
    • Une source de données pour produire plusieurs jeux de données Power BI.
  • Plusieurs sources -> plusieurs jeux de données ->un espace de travail de jeu de donnée (WS-DS)
    • Plusieurs jeux de données dans le même WS-DS, espace de travail de jeux de données.

Une source de données pour un jeu de données Power BI​

Configuration la plus commune et considérer comme le standard ou le mode par défaut. CE scénario

Pour chaque sujet d'affaires, un espace de travail est créé pour y publier le modèle de données Power BI.

Cette configuration est la configuration par défaut. Les projets de développement corporatifs, développés à partir de la voute de données, sans contexte de sécurité ni contexte d'affaires, suivent cette configuration.

  1. Source de données = Comptoir

    Les sources de données valides sont :

    • Comptoir de données EODE.
    • Comptoir de données projet Fast-Track.
    • Sandbox EODE.

  2. Serveur de Gateway

    • Requis pour rafraichir les données sur site.

  3. Paramètre de passerelle

    • Sert à quoi :
    • Utiliser des comptes de service pour
    • spécifique à chaque source de données pour chacun des paliers de développement.

  4. WS-DS: Espace de travail de jeux de données

    • Le jeu de données est déployé dans son espace de travail spécifique.

    • Valider si le jeu de données doit être certifié.

    • Recommande un seul jeu de données par espace de travail.

    • Si plusieurs jeux de données sont déployés dans un même espace de travail:

      • Les regrouper par sujet affaires (devrait être un cas d'exception) ;
      • Valider que les accès sont et seront les mêmes pour tous les jeux de données. Dans le doute, séparer les jeux de données dans des espaces de travail différents.

Configuration plusieurs sources de données​

Dans le cas où deux sources de données sont utilisées pour créer un jeu de données Power BI, il n'y a aucun impact sur l'architecture fonctionnelle du service Power BI.

Un seul jeu de données et un seul WS-DS sont créés.

Solution recommandée pour intégrer différents sujet affaires dans un même jeu de données Power BI.

  1. Au moins deux sources de données de types comptoir sont requises pour la conception du modèle.

    Les sources de données valides sont :

    • Comptoir de données EODE = DM ;
    • Comptoir de données projet Fast-Track = DML ;
    • Sandbox EODE = SBX.

  2. Serveur de Gateway

  3. Paramètre de passerelle

    • Toutes les sources utilisées dans le modèle Power BI doivent être configurées dans la passerelle pour configurer des rafraichissements ou utiliser le mode de connexion direct-query

  4. WS-DS: Espace de travail de jeux de données

    Le jeu de données sera créer à partir de plusieurs sources différentes ;

    Le fichier Power BI Desktop doit se connecter sur chacune des sources ;

    Le modèle Power BI doit harmoniser les sources ou cela doit se faire dans un comptoir dédié à cet effet.

Configuration plusieurs jeux de données Power BI​

Générer plusieurs jeux de données Power BI à partir d'une même source de données.

  1. WS-DS-1 : Espace de travail de jeux de données 1
    • Développer un jeu de données et le publier dans son propre espace de travail
  2. WS-DS-2 : Espace de travail de jeux de données 2
    • Développer un second jeu de données et le publier dans son propre espace de travail

Même si la source est la même, lorsque plusieurs jeux de données sont créés, on les publie dans des espaces de travail dédiés.

  • Cela permet de découpler l'accès à un jeu de données de la source de données sur lequel il est créé.
  • Cela permet de gérer facilement la sécurité par jeu de données par la suite.
    • Évolution des besoins d'affaires ;
    • Évolution des contextes de sécurité ;
    • Ajout de libre-service ;
    • Donner accès à des concepteurs affaires.

CONFIGURATION PLUSIEURS JEUX DE DONNÉES DANS UN MÊME ESPACE DE TRAVAIL

Espace de travail de jeu de données composé de plusieurs jeux de données Power BI. La source de ces jeux de données peut être différente.

  1. et 2. sur site : La configuration de chacune des sources se fait comme pour la configuration standard.

  2. WS-DS: Espace de travail de jeux de données

    • Un seul espace de travail de jeux de données est créé pour plusieurs jeux de données.

  3. Cette solution est une alternative à des situations d'exceptions.

    • Si un projet utilise plusieurs sources de données qui sont uniquement utilisées pour ce projet.
    • Tous les développeurs, concepteurs et supers utilisateurs impliqués ont accès à tous les jeux de données.
    • Le nombre de sources rend la gestion des espaces de travail ingérable.
      • Dans certain cas, il peut-être utile de regrouper des jeux de données dans un même espace de travail et de gérer les accès en bloc.
