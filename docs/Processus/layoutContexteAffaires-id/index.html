Architecture fonctionnelle - Contexte d'affaires

ARCHITECTURE FONCTIONNELLE - CONTEXTE D'AFFAIRES

La séparation des groupes n'est pas lié à une contrainte de sécurité.

Chaque groupe de rapports, ou contexte d'affaires, se concrétise sous la forme d'une application Power BI.

Cela implique:

Option #2 de dessin: Myriam tu en penses quoi, avec tous les no ou moins ?

Mise en place

  1. Configurez les sources et jeux de données

    • Aucun impact .
    • Configurer normalement comme un projet affaires EODE (ajouter le lien vers la page projet affaire EODE : PBI_Arch_Fnct_31_Layout défaut.md /Projet affaires EODE (défaut))

  2. Créer un WS-RPT, espace de travail de rapport, pour chaque contexte d'affaire

  • Assigner le groupe AD de développeur EODE comme administrateur de l'espace de travail.

  • Personne d'autre ne doit être administrateur des espaces de travail (groupe AD ou ressource).

  1. Créer un groupe AD pour d'application Power BI, pour chaque contexte d'affaires.

  • Assigner le groupe AD sur:

    • L'application Power BI correspondant au contexte d'affaires

    • Le jeu de données dans le WS-DS avec le privilège Lecture.

  1. Créer le WS-LB, l'espace de travail libre-service, pour chaque contexte d'affaires.

  • Assigner le groupe de développeur EODE comme administrateur de l'espace de travail.

  • Créer un groupe AD de tâches super utilisateur pour le sujet du libre-service.

  • Assigner le groupe AD de tâches super utilisateur sur:

  • L'espace de travail libre-service WS-LB avec le privilège de membre.

  • Le jeu de données dans le WS-DS avec le privilège Build.

  1. Gérer les accès à l'application libre-service (si requis).

    • Assigner le groupe AD de tâches super utilisateur sur l'application libre-service.

      Créer un groupe AD pour l'application Power BI (si requis)

      Voir [Point de validation](#####Point de validation) dans l'option libre-service

      Créer un lien vers : Voir PBI_Arch_Fnct_31_Layout défaut / section Gérer les accès à l'application libre-service (_si requis). pour plus de détail_

  2. Créer un seule groupe de tâche AD de libre-service

    • Les Super-utilisateurs ont accès à 100 % des données.

    • Le même groupe de super utilisateur peut créer les rapports libre-service pour tous les contexte.

      Peut importe le nombre de contexte d'affaires, il n'est pas requis de créer des groupes AD de libre-service différent pour chacun des contextes.

  3. Créer les groupes d'application Power BI, mêmes utilisateurs.

    • La première option est que les mêmes utilisateurs consomment l'ensemble des contextes d'affaires.

    • Dans cette situation ont créé un groupe AD de tâche qui représente les utilisateurs et le sujet.

    • Assigner le groupe de tâche de consultation sur toutes les applications et pour tous les contextes d'affaires

      Ne pas oublier que les contextes d'affaires n'ont pas pour but de limiter l'accès aux données.

  4. Créer les groupes d'application Power BI, utilisateurs différents

    • La seconde option est que des utilisateurs consomment différent groupe de rapport ou contextes d'affaires.
    • Dans cette situation, il est préférable de créer des groupes AD d'application qui correspondent à chaque contexte d'affaires.
    • Assigner le groupe d'application sur le contexte d'affaires correspondant.

  5. Gérer les contextes d'affaires supplémentaires

    • Répéter autant que nécessaire les étapes 1 à 9.
    • Soyer critique lors de l'analyse du découpage en contexte d'affaires:
    • Faire l'effort de comprendre pourquoi les gens d'affaires veulent séparer les rapports.
    • Avoir trop de contexte peux devenir complexe à gérer pour les développeurs.
    • L'objectif n'est pas de faciliter le développement, mais de simplifier la vie des utilisateurs.
    • Avoir trop de contexte peux devenir complexe pour les utilisateurs pour se retrouver.
    • Le découpage doit être clair et logique du point de vue des utilisateurs.
