ARCHITECTURE FONCTIONNEL POWER BI - CONTEXTE DE SÉCURITÉ

Contexte de sécurité sur les rapports​

Cette configuration s'applique lorsque la sécurité affecte l'accès aux rapports, mais pas la source de données. En d'autres termes, la même source de données est utilisée pour créer plusieurs contextes de sécurité.

Cela implique :

  • Le contexte de sécurité s'applique au niveau de la consommation de l'information et non au niveau de la conception des jeux de données et rapports.
  • Les développeurs de chacun des contextes ont accès à 100% de la source de données.
    • Points 1-2-3 sur l'image.
  • Les utilisateurs ne peuvent pas consulter 100% des données/informations.
    • Point 6 sur l'image. Chaque groupe d'utilisateur a accès a un sous ensemble de rapports.

En générale, la sécurité limitera la consultation de certaines tables de fait pour certains groupes d'utilisateurs.

  • Il est alors possible de créer plusieurs jeux de données Power BI en limitant le contenu à ce qui est autorisé dans un contexte de sécurité. (point 4 sur image)

Mise en place

  1. Source de données

    • Aucun impact sur la source de données.

    • On ne gère pas la sécurité au niveau de la source de données.

  1. Serveur de Gateway Power BI

    • Aucun impact sur le serveur de Gateway.

  1. Configuration de la passerelle

    • Aucun impact sur la configuration de la passerelle.

    • La configuration de passerelle se fait normalement en utilisant un compte de service.

  1. Créer un WS-DS, espace de travail de jeu de données, pour chaque contexte de sécurité.

    • La sécurité des contextes est gérée au niveau des espaces de travail WS-RPT et au niveau des jeux de données.

    • Le jeu de données créé pour chaque contexte est publié dans le WS-DS correspondant.

    • Assigner le groupe AD de développeur EODE comme administrateur de l'espace de travail.

      Personne d'autre ne doit être administrateur des espaces de travail (groupe AD ou ressource).

  2. Créer un WS-RPT, espace de travail de rapport, pour chaque contexte de sécurité.

    • Assigner le groupe AD de développeur EODE comme administrateur de l'espace de travail.

      Personne d'autre ne doit être administrateur des espaces de travail (groupe AD ou ressource).

    • Créer un groupe de sécurité pour l'application, pour chaque contexte de sécurité.

    • Assigner le groupe de sécurité AD de l'Application sur l'application du contexte correspondante.

Contexte de sécurité sur les jeux de données​

Cette configuration s'applique lorsque la sécurité limite la consultation directement au niveau de la source. Non seulement les utilisateurs n'ont pas accès à 100% des données sources, mais les développeurs et super-utilisateurs sont aussi affectés par le contexte de sécurité.

Par exemple, voir les salaires sur le sujet affaires ressource humaine.

Ce implique:

  • Au niveau de la source, il y a un API, sous forme de vues SQL, qui est mis en place et sécurité pour limiter l'accès à la consultation par les développeurs.
  • Chacun des contextes de sécurité aura sa propre configuration de passerelle et apparaitra comme une source distincte pour le développeur.
  • Dans les faits, cela n'implique pas que la base de données source, le comptoir, soit physiquement différente pour chacun des contextes.
    • Cependant, il est possible que dans certains cas, une source de données distincte soit requise et qu'un comptoir de donnée différent soit mis en place (point 7 sur image).

Mise en place

  1. Source de données

    • API de contexte sur la source de données.

      • Des vues SQL sécurisées pas contexte limites l'accès aux données source.

        • Chaque groupe de vues est créé sous un schéma de base de données qui précise le contexte de sécurité.

      • Créer un groupe de tâche AD de développement pour le / les contextes

      • Assigner le groupe de tâche AD de développement sur les schémas correspondant dans le comptoir

  1. Serveur de Gateway Power BI

    • Aucun impact sur le serveur de Gateway.

  1. Configuration de la passerelle

    • Créer une configuration de passerelle pour chaque contexte de sécurité.

    • La configuration de passerelle se fait normalement en utilisant un compte de service.

  1. Créer le WS-RPT, l'espace de travail de rapport pour chacun des paliers (INT-ACC-PRD), pour chaque contexte de sécurité.

    • La sécurité des contextes est gérée au niveau des espaces de travail WS-RPT et au niveau des jeux de données.

    • Le jeu de données créé pour chaque contexte est publié dans le WS-DS correspondant.

    • Assigner le groupe AD de développeur EODE du contexte comme administrateur de l'espace de travail.

      Personne d'autre ne doit être administrateur des espaces de travail (groupe AD ou ressource).

  2. Créer les WS-RPT, l'espace de travail de rapport pour chacun des paliers (INT-ACC-PRD), pour chaque contexte de sécurité.

    • Assigner le groupe AD de développeur EODE du contexte comme administrateur de l'espace de travail.

      Personne d'autre ne doit être administrateur des espaces de travail (groupe AD ou ressource).

  3. Créer un groupe AD les applications Power BI, pour chacun des paliers (INT-ACC-PRD) et pour chaque contexte de sécurité.

    • Créer un groupe de sécurité d'application, pour chaque contexte de sécurité.
    • Assigner le groupe AD d'application sur:
      • L'application de rapport correspondant au contexte de sécurité.
      • Le jeu de données dans le WS-DS correspondant au contexte de sécurité, avec le privilège Lecture.

Contexte de sécurité avec libre-service​

Cette configuration s'applique lorsque:

  1. Un contexte de sécurité est présent et que l'option libre-service est requise.
  2. Des super utilisateurs font partie du projet.

  1. Configurer la source

    • Selon le contexte, consulter [Contexte de sécurité sur les rapports](about:blank##Contexte de sécurité sur les rapports) ou [Contexte de sécurité sur les jeux de données](about:blank##Contexte de sécurité sur les jeux de données) pour mettre en place la ou les sources de données.

  1. Créer les WS-LB, espace de travail libre-service en production (PRD), pour chaque contexte de sécurité.

    • Assigner le groupe de développeur EODE du contexte comme administrateur de l'espace de travail libre-service.
    • Créer un groupe AD de tâche super utilisateur pour chaque contexte de sécurité de libre-service.
    • Assigner le groupe AD de tâche super utilisateur du contexte de sécurité sur:
      • L'espace de travail libre-service WS-LB, du contexte de sécurité correspondant, avec le privilège de membre.
      • Le jeu de données dans le WS-DS, du contexte de sécurité correspondant, avec le privilège Build.

  1. Gérer les accès à l'application libre-service (si requis).

    • Assigner le groupe AD tâche super utilisateur sur l'application libre-service.

    Devrait être un lien vers la page PBI_Fnct_31_Layout défaut / Point de validation

    Point de validation​

    Qui doit avoir accès aux rapports libre-service

    • Tous le monde : utiliser le groupe AD de l'application WS-RPT.

    • Les super utilisateurs seulement : Utiliser le groupe AD Super utilisateur Libre-service pour gérer l'application.

    • Peux de gens et ils sont différent des supers-utilisateur : Créer un groupe AD spécifique pour l'application libre service.

    • Au besoin: Créer un groupe AD pour l'application libre-service.

    • Assigner le groupe AD de l'application libre-service sur:

      • L'application de libre-service
      • Le jeu de données dans le WS-DS avec le privilège Lecture.

    • Au besoin: Assigner le groupe AD de l'application des rapports WS-RPT sur l'application de libre service optionnel)

      • Cela permet aux utilisateurs de l'application de rapports officiels, de consulter les rapports ad-hoc.
      • Cet action remplace généralement la création d'un groupe AD spécifique pour gérer l'application de libre-service
      • Faire seulement si tous les utilisateurs peuvent et doivent consulter les rapports de libre-service.
