Architecture fonctionnelle par type de projet

Architecture fonctionnelle par type de projet

Ce document présente les configurations de l'architecture fonctionnelle du service Power BI les plus fréquentes et standards.

  • Le dessin présente la configuration à mettre en place dans le service Power BI pour le sujet.
  • Le texte sous les dessins précise et explique le choix de cette configuration.
  • Les opérations requises pour la mise en place des différentes composantes sont détaillées sous le sujet de documentation : lien vers pratiques Power BI.

Référez-vous à la documentation de nomenclature pour:

  • Nom d'espace de travail : Comment nommer vos espaces de travail lors de la création (lien vers : Nomenclature entité Power BI... à venir).

  • Groupe AD : Comment nommer vos groupes de sécurité (lien vers : Nomencalture groupe AD... à venir).

Projet affaires​

Scénario typique: Projet affaires dont la conception est réalisée par l'équipe corporative. Les données sources proviennent de l'entrepôt de données. Le public cible consommera des rapports normalisés avec des données validées et fréquemment rafraichis. Cas typique de données et rapport certifié.

  1. Configuration des paramètre de paserelle

    • La passerelle permet d'automatiser le rafraichissement des données et d'utiliser les modes de connexions "live query" et "direct query";

    • Si vous utilisez une nouvelle source de données, un nouveau comptoir doit être mis en place ou des ajouts doivent être intégré à un comptoir existant.

    • La gestion des source de données se fait dans le service Power BI, par la console de gestion des paramètres de passerelle .

  1. WS-DS, l'espace de travail de jeux de données est utilisé exclusivement pour publier le jeu de données correspondant à un sujet d'affaire spécifique.

    • Son rôle est de permettre le partage d'un jeu de données à plusieurs concepteurs de rapports.
    • Cela permet d'avoir un seul jeu de données pour un sujet spécifique.
    • Un espace de travail WS-DS-sujet est créé pour chacun des paliers (INT-ACC-PRD);
    • Ce type d'espace de travail est dédié aux jeux de données et est accessible seulement par l'équipe de développement corporative.
      • Aucun rapport n'est publié dans ce type d'espace de travail;
    • On donne accès au jeu de données, mais pas à l'espace de travail.
    • L'application Power BI n'est pas créer pour ce type d'espace de travail.

  1. WS-RPT, l'espace de travail de rapport est utilisé pour publier les rapports en lien avec le sujet affaires du WS-DS-sujet

    • Un espace de travail de rapport WS-RPT est créé pour chaque palier (INT-ACC-PRD);

    • Ce type d'espace de travail est dédié aux rapports officiel et développé par l'équipe corporative;

    • Il est accessible seulement par l'équipe de développement corporative;

  1. L'application Power BI

    • Le parage des rapports se fait par l'application;
    • Les utilisateurs ont accès à l'application grâce à un groupe AD.

Projet affaires avec option libre-service​

Scénario typique: Comme pour le projet affaires EODE, mais avec des super utilisateurs dans le projet. Les super utilisateurs ont accès au jeu de données pour créer des rapports et les partager avec des collègues. Le libre-service est optionnel et est mis en place uniquement lorsque l'équipe affaires dispose de ressources possédant les compétences de super utilisateurs ou désirant monter en compétence pour remplir ce rôle.

  1. Mettre en place une configuration projet affaires EODE

  2. Configurer le jeu de données partagé

    • Dans l'espace de travail WS-DS du jeu de données, configurez le jeu de données en mode partagé.

  1. Créer le WS-LB, l'espace de travail libre-service en production (PRD).

    • Assigner le groupe de développeurs EODE comme administrateur de l'espace de travail.

    • Créer un groupe AD de tâches super utilisateurs pour le sujet du libre-service.

    • Assigner le groupe AD de tâches super utilisateurs sur:

      • L'espace de travail libre-service WS-LB avec le privilège de membre.

      • Le jeu de données dans le WS-DS avec le privilège Build.

  1. Gérer les accès à l'application libre-service (si requis).

    • Assigner le groupe AD tâches super utilisateurs sur l'application libre-service.
    Point de validation​

    Qui doit avoir accès aux rapports libre-service

    • Tout le monde : utiliser le groupe AD de l'application WS-RPT.

    • Les super utilisateurs seulement : Utiliser le groupe AD Super utilisateurs Libre-service pour gérer l'application.

    • Peu de gens et différents des super utilisateurs : Créer un groupe AD spécifique pour l'application libre-service.

    • Au besoin: Créer un groupe AD pour l'application libre-service.

    • Assigner le groupe AD de l'application libre-service sur:

      • L'application de libre-service ;
      • Le jeu de données dans le WS-DS avec le privilège Lecture.

    • Au besoin: Assigner le groupe AD de l'application des rapports WS-RPT sur l'application de libre-service (optionnel)

      • Cela permet aux utilisateurs de l'application de rapports officiels, de consulter les rapports ad hoc.

      • Cette action remplace généralement la création d'un groupe AD spécifique pour gérer l'application de libre-service

        Attention

      • Faire seulement si tous les utilisateurs peuvent et doivent consulter les rapports de libre-service.

        :::

Configuration libre-service seul​

Scénario typique: Une équipe de super utilisateurs souhaite exploiter un jeu de données pour collaborer à la création d'analyses, de rapports et possiblement les partager avec des collègues.

Ce scénario implique que l'équipe de développement corporatif crée le modèle, mais ne produit aucun rapport officiel. C'est l'équipe de super utilisateurs qui est responsable de 100% de la réalisation des rapports.

  1. Configurer la passerelle - Optionnel

    • Si vous utilisez une nouvelle source de données, vous devez créer un nouveau comptoir.

    • Dans la console de gestion des paramètres de passerelle, ajoutez une configuration pour votre nouvelle source de données.

  1. Créer le WS-DS, l'espace de travail de jeux de données pour chacun des paliers (INT-ACC-PRD)

    • Assigner le groupe AD de développeurs EODE comme administrateur de l'espace de travail

    • Personne d'autre n'a accès à cet espace de travail (groupe AD ou ressources)

  1. Créer le WS-LB, l'espace de travail libre-service en production (ACC-PRD).

    • Assigner le groupe de développeurs EODE comme administrateur de l'espace de travail.

    • Créer un groupe AD de tâches super utilisateurs pour le sujet du libre-service.

    • Assigner le groupe AD de tâches super utilisateurs sur:

      • L'espace de travail libre-service WS-LB avec le privilège de membre.

      • Le jeu de données dans le WS-DS avec le privilège Build.

Créer un groupe AD pour l'application Power BI (si requis)

Voir [Point de validation](#####Point de validation) dans l'option libre-service

Configuration conception affaires​

Scénario typique: Un ou des concepteurs désirent explorer des données de manière autonome.

  1. Configurer la passerelle

    • Dans la console de gestion des paramètres de passerelle, ajoutez une configuration pour votre nouvelle source de données.

    • Normalement cela sera une base de données nommée SBXU_sujet.

  1. Créer le WS-SBXU, l'espace de travail de jeux de données pour les paliers d'acceptation et production (ACC - PRD)

    • Assigner le groupe AD de développeurs EODE comme administrateur de l'espace de travail ;
    • Créer un groupe AD de concepteurs affaires ;
    • Assigner le groupe AD de tâches concepteurs affaires sur :
      • L'espace de travail de conception affaires WS-SBXU avec le privilège de membre.

Configuration exploration de développement (Sandbox)​

Scénario typique: Spécifiquement créé pour l'équipe de développement corporatif, cette configuration permet d'explorer des solutions pour valider la faisabilité de projet. Preuve de concept, exploration de données, conception ou recherche et développement (R&D) de solutions techniques sont différentes options orientées vers cette configuration.

Configuration de plusieurs sujets affaires pour un même public utilisateur​

Scénario typique: Ce scénario est présenté pour gérer un cas d'exception du scénario de projet affaires EODE. Il peut arriver que les utilisateurs aient à consommer des rapports provenant de plusieurs sujets affaires et que leurs contextes d'affaires impliquent que ces rapports soient consultés dans la même application Power BI.

Bien qu'il soit possible de mettre plusieurs jeux de données de sujets différents dans un même WS-DS (espace de travail de jeu de données), nous ne recommandons pas cette solution.

  1. Créer le WS-RPT, l'espace de travail de rapport pour chacun des paliers (INT-ACC-PRD)
