Principe de découpage fonctionnel

Principe de découpage fonctionnel

  1. Le découpage fonctionnel se fait en premier par contexte d’affaires.

    • Le sujet d’analyse définit le périmètre du jeu de données.
    • Le public cible définit le périmètre de la publication des rapports.

    Exemple:

    Pour le sujet d'affaires analyse des fonds, nous avons créé plusieurs rapports différents (>15). Certains de ces rapports sont :

    • consommés trimestriellement lors de la révision des résultats avec les partenaires ;
    • consommés quotidiennement pour faire des analyses ;
    • consommés spécifiquement dans le processus d'analyse du type de fonds XYZ.

    Présenter tous les rapports dans une même application n'est pas efficace et crée de la confusion chez les utilisateurs.

    Les utilisateurs demandent donc de séparer les rapports en fonction du contexte dans lequel ils utilisent les rapports.

    On crée alors 3 contextes d'affaires :

    • Sommaire trimestriel ;
    • Analyse des fonds ;
    • Processus XYZ.

  1. Les jeux de données et les rapports, pour un même sujet affaires, ne sont pas dans le même espace de travail

    • Exception pour le développement exploratoire et le développement de prototype. Dans ces cas spécifiques, on utilise le même espace de travail pour le jeu de données et les rapports.

  1. Les jeux de données ont un espace de travail dédié pour chacun des sujets affaires ou domaines d’affaires.

    • Des jeux de données sur des sujets différents et qui ne sont pas utilisés par les mêmes ressources ne sont pas dans les mêmes espaces de travail.
    • Des jeux de données sur des sujets différents, mais toujours consommés par les mêmes ressources peuvent être dans le même espace de travail.
      • On recommande fortement de créer des espaces de travail spécifiques pour chacun des sujets.
      • Cela permet une meilleure évolution lors d’un ajout de données et pour gérer les contextes de sécurité.

  1. Les rapports ont un espace de travail dédié pour chacun des sujets ou domaines d’affaires.

    • Les rapports sont développés sur un jeu de données partagé :

      Jeu de données partagé :

      Le fichier .pbix des rapports ne contient pas le jeu de données, mais se connecte en mode « Live connection » sur un jeu de données publié dans le service Power BI.

    • Si un même jeu de données est utilisé pour créer des rapports pour 3 contextes d’affaires différents, 3 espaces de travail de rapports seront créés.

  1. Un espace de travail de rapport supplémentaire est créé pour le libre-service | Optionnel

    • Les rapports créés par les super-utilisateurs ne sont pas publiés dans le même espace de travail que les rapports faits par l’équipe de développement BI. Cela permet de conserver le contrôle sur les modifications apportées aux rapports officiels et corporatifs.

    L’espace de travail de libre-service pour un sujet affaires est optionnel*.

    • Ce n’est par tous les sujets affaires qui nécessitent du libre-service ou disposent de super-utilisateurs.
    • Au même titre, un espace de libre-service peut exister sans qu’il n'y ait d’espace de rapport corporatif pour le même sujet. Il est possible que certains jeux de données soient créés pour un usager en libre-service exclusivement.
    • Dans ce cas, les rapports sont tous créés par les super-utilisateurs. Seulement un espace de travail libre-service est créé (le jeu de données a quand même son propre espace de travail).

  1. Un espace de travail de rapport supplémentaire est créé pour chacun des contextes affaires | Optionnel

    Cela affecte les espaces de travail de :

    • Rapport
    • Libre-service

  1. Un espace de travail de rapport supplémentaire est créé pour chacun des contextes de sécurité | Optionnel

    • Dans le cas ou des impératifs de sécurité limitent l’accès aux données pour un même sujet d’affaires, un second jeu de données sera créé pour contenir uniquement les données sources répondant aux exigences de sécurité.
    • Cela affecte les espaces de travail de :
      • Jeu de données
      • Rapport
      • Libre-service
    • **Optionnel**
      • Les espaces de travail liés au contexte de sécurité sont optionnels, car seulement présents lorsque la sécurité l’exige.

    Exemple:

    Pour le sujet d'affaires ressources humaines, plusieurs types d'informations sont disponibles:

    • Profil de l'employé ;
    • Informations personnelles : adresse, tél., courriel, etc. ;
    • Informations d'organisation : tâche, directeur, département, etc. ;
    • Salaires ;
    • Dossier médicale.

    Certaines de ces informations sont classées à haut privilège. Les gens d'affaires nous informe que seulement les ressources avec un privilège élevé peuvent accéder aux informations de salaire et médicale.

    Cela implique deux contextes. Comme cela est lié à l'accès aux données pour des raisons de sécurité, cela en fait des contextes de sécurité.

    On crée alors 2 contextes de sécurité :

    • RH (Ressources humaines)
    • RH_accès limité

    On crée également des groupes de sécurité (AD) qui seront utilisés pour limiter les accès au jeu de données.
