Architecture fonctionnelle - jeu de données

ARCHITECTURE FONCTIONNELLE POWER BI: JEU DE DONNÉES

Scénarios possibles:​

La configuration à mettre en place dans le service Power BI pour gérer les sources de données dépend de la relation entre la source et le jeu de données Power BI.

  • Une source / un sujet affaires -> un jeu de données (défaut).
  • Plusieurs sujets affaires -> un jeu de données.
  • Une source / un sujet affaires -> plusieurs jeux de données Power BI.
  • Un espace de travail pour plusieurs jeux de donnée Power BI.

Composants requis pour instancier les solutions​

Chacune des sous-sections, disponibles par les liens de la section précédente [scénarios possibles](about:blank##Scénarios possibles), présentent en détail les variantes propres à chaque scénario.

En fonction des scénarios, certains paramètres seront adaptés, mais voici les composants utilisés pour mettre en place le scénario d'un sujet affaires -> un jeu de données Power BI.

Ce scénario est le plus simple, le plus fréquent et est identifié comme le scénario par défaut. On l'utilise ici pour présenter le rôle des composants.

Image manqante JEFF trouve là

  1. Source de données

  • Utilisé comme source de données par le jeu de données Power BI.

  • C'est une base de données de type comptoir de données.

    • Sur site, c'est une base de données SQL

      ou

    • Dans l'infonuagique, c'est une base de données Snowflake.

  1. Serveur de passerelle
  • Requis pour rafraichir les données sur site.
  • Sert de passerelle entre les données sur site et le service Power BI dans l'environnement infonuagique.
  1. Paramètre de passerelle
  • Interface de configuration des paramètres spécifiques pour une source de données, dans le service infonuagique Power BI.
    • Nom de la passerelle.
    • Nom du serveur sur site.
    • Nom de la base de données.
    • Comptes de services et mots de passe pour accéder à la source.
  • Il est nécessaire de configurer une fois chacune des sources de données dans les paramètres de passerelle pour gérer la fréquence de rafraichissement.
  • Utiliser un compte de service spécifique pour chacune des sources à configurer.
  1. WS-DS : Espace de travail de jeu de données
  • L'espace de travail de jeu de données est instancié dans le service Power BI et sert de dépôt pour le jeu de données.
  • La sécurité sur l'espace de travail limitera l'accès aux données sans avoir à mettre en place de sécurité particulière au niveau de la source de données (BD SQL ou Snowflake).
  1. Jeu de données Power BI
  • Produit avec Power BI Desktop et publier dans le service infonuagique Power BI.
  • Structure de données tabulaire sous forme de modèle étoile exposée au développeur, concepteur affaires et super-utilisateur pour produire des rapports.
  • Sert de base de données (tabulaire) pour la production des rapports et tableaux de bord.

Source de données acceptées pour alimenter un jeu de données Power BI​

Pour le développent corporatif​

La seule source de données valide est un comptoir.

Lien vers la norme couche & sous-couche : à venir

Il existe plusieurs types de comptoirs de données.

Comptoir de données

  • Produit par l'équipe BI, pour les projets affaires.
  • Préfixe de base de données = "DM_"

Comptoir de données et projet Fast-Track

  • Produit par l'équipe BI, pour les projets rapides ou temporaires non intégrés dans la voute de données.
  • Préfixe de base de données = "DMF_"

Sandbox DGVI

  • Pour les projets d'exploration, de validation et d'évaluation de solution.
  • Préfixe de base de données = "SBX_"

Sandbox d'exploration affaires

  • Pour les projets de développement en milieu utilisateur.
  • Préfixe de base de données = "SBXU_"

Pour le développement en milieu utilisateur​

En plus des comptoirs, les concepteurs affaires peuvent utiliser Excel et des fichiers plats comme source de données.

Autres sources de données, développement​

Dans les cas où la source de données n'est pas un comptoir, il faut la rendre disponible par un comptoir de données.

Exemple:

Pour explorer le Data Lake, la voute de données brute et la voute affaires, un Sandbox sera créé afin de rendre les sources requises disponibles aux développeurs et/ou concepteurs.

Pour explorer des fichiers plats, il est recommandé de charger la source dans un Sandbox et d'utiliser ce Sandbox comme source pour produire le jeu de données Power BI.

Objectif d'utiliser un comptoir pour consommer les différentes données provenant des couches de l'entrepôts de données.

  • Permet de ne pas donner accès aux sous-couches de données, autres que la couche des comptoirs.
  • Contrôler les données accessibles aux concepteurs par le comptoir.
    • Le concepteur a seulement accès aux données présentes dans le comptoir, mais pas à la source originale des données.
  • Permet de limiter et d'encadrer l'accès aux données d'une couche de l'entrepôt sans avoir à sécurisé cette couche par accès utilisateur.
    • Le Data Lake peut contenir des centaines de tables provenant de plusieurs sources de données.
    • Un comptoir peut exposer un sous-ensemble du Data Lake pour rendre disponibles les tables d'un sujet donné.
    • L'accès aux concepteurs sera donné sur le comptoir, mais pas sur le Data Lake, limitant automatiquement l'accès aux données à celles exposées dans le jeu de données Power BI.
  • Personne n'a accès directement au Data Lake et à la voute (brute et affaires).
    • Le "Cross chaining" des droits d'accès SQL, permet de créer une vue dans un Sandbox sur des données du Data Lake et de donner accès en lecture à cette vue, dans le Sandbox, sans donner accès au Data Lake.
  • La gestion des accès utilisateurs se fait sur une seule couche de l'entrepôt.
    • La couche des comptoirs.
    • Aucune autre couche n'est exposée à des utilisateurs autre que les développeurs de l'équipe BI.
